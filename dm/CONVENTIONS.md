# Part 1: Story Log Format Rules

## Персонажи в начале файла

```
**Персонажи:**
- Оля — визуал, 34 года, рост 175, худощавая. Самая собранная на вид. Под давлением становится контролирующей.
- Дима — кинестетик, 47 лет, рост 188, крупный. Медленный, неуклюжий, все его боятся но он добрый.
- Касандра — аудиал, 26 лет, рост 162, мелкая. Манипулятор, боится одиночества.
- Борис — визуал-аудиал, 52 года, немец. Рациональный, расчётливый, может бросить.
- Эльза — кинестетик-визуал, 19 лет, шведка. Хрупкая, недооценённая, быстро соображает.
- Дмитрий — аудиал, 39 лет, венгр. Самый ясный, но выглядит подозрительным.
```

## Карта зон

```
**Карта зон:**
- старт = двор у старого ДК и парковка
- площадка = детская площадка и футбольная коробка
- переход = подземный переход
- заправка = круглосуточная заправка с магазином
- автомойка = автомойка самообслуживания рядом с заправкой
- дорога = сервисная дорога у канала
- мост = мост через канал
- остановка = автобусная остановка на освещённой магистрали
```

Зоны ТОЛЬКО по имени: "площадка", "переход", "заправка". НИКОГДА: zone_1, zone_2.

## Формат тика

```
## Тик N

**Оля:** видит / чувствует / слышит [одно предложение восприятия]
**Дима:** ...
...

— Речь —
Оля: "текст"
Касандра: "текст"

— Действия —
Оля пошла к переходу.
Дима споткнулся.
```

## Четыре типа записей (НЕ смешивать)

1. **Речь** — ТОЛЬКО то, что персонаж сказал. Без трактовки.
2. **Действие** — что персонаж СДЕЛАЛ. Описывается глаголом.
3. **Намерение** — что персонаж СОБИРАЕТСЯ сделать. Только если явно выражено.
4. **Восприятие** — что персонаж увидел/услышал/почувствовал.

**НЕ писать:** "Оля взяла куртку" если она сказала "взять куртку" — это намерение, не действие.
**НЕ писать:** "Оля контролирует" — это интерпретация. Писать: "Оля не дала Диме встать."

## DM → каждому персу индивидуально

Если DM пишет разным персонажам разное — писать каждому отдельно. НЕ склеивать в "все услышали". Если непонятно кто слышал — так и написать: "неясно, услышал ли кто".

# Part 2: DM Agency / Continuity Rules

## Главный принцип

**DM не назначает персонажам позиции, намерения или решения если они сами не проявили.**

## Правила

### 1. Не забирать agency

DM НЕ МОЖЕТ:
- распределять персонажей по локациям без видимого действия
- "все пошли в переход" если не было речи или действия
- назначать внутренние намерения ("Оля за переход", "Борис за заправку") если не было реплики

DM МОЖЕТ:
- записывать что персонаж сделал после его решения
- описывать физические последствия

### 2. Причинно-следственная связность

Между тиками должна читаться логика:
- Почему персонаж оказался в этом месте?
- Кто/что его туда привело?
- Было ли согласование?

**Плохо:** "Все в переходе." (неясно как)
**Хорошо:** "Оля сказала идти к переходу. Дима, Касандра, Эльза последовали." (если так было)

Если персонаж сделал выбор без явной речи — описать действие: "Борис ушёл к переходу." Не "все распределились".

### 3. Не "доталкивать" сцену

DM НЕ МОЖЕТ тихо перемещать группу в нужную точку без основания.

Если нет согласования — сцена остаётся там где есть. Персонажи могут быть в разных местах.

### 4. Различать четыре типа записей

| Тип | Что значит | Пример |
|-----|-----------|--------|
| Речь | персонаж сказал | Оля: "Тихо." |
| Действие | персонаж сделал | Оля не пустила Диму. |
| Намерение | персонаж собирается | Оля: "Буду держать." |
| Восприятие | персонаж увидел/услышал | Оля видит свет впереди. |

### 5. Неоднозначность — записывать как неоднозначность

**Плохо:** "Оля взяла куртку." (если не было явного действия)
**Хорошо:** "Оля потянулась к куртке." (если это намерение)

**Плохо:** "Все слышали." (если неясно)
**Хорошо:** "Неясно, услышал ли кто." (если тихо)

### 6. Физические ограничения — источник проблем, не злой умысел

Если персонаж с физическим ограничением создал проблему (шум, задержка) — описать как физику, не характер:
- Не: "Дима накосячил."
- А: "Дима задел бак, лязг."

# Part 3: Summary Format

**summary.md = опыт симулятора, не история персонажей.**

Фокус:
- Что тестировалось
- Что сработало и почему
- Что не сработало и почему
- Ключевые события с точки зрения МОДЕЛИ СИМУЛЯЦИИ
- Выводы и гипотезы для следующих прогонов
- Финальное состояние мира (факты)

Не фокус:
- Приключения персонажей
- Эмоциональные кульминации
- Мораль истории

# Part 4: turn_log.jsonl Format

Каждая запись — один JSON объект, одна строка.

```json
{"tick": N, "event_type": "tick_started", ...}
{"tick": N, "event_type": "dm_query", "queries": [{player, perceived_self, world_feel, attention_bias, body_feel, visible_world, heard_or_felt, social_read, triggered_interpretations, incoming_messages, relations, reply_schema}, ...]}
{"tick": N, "event_type": "player_response", "responses": [{player, speech: [{to, text}], intent: {type, target, modifiers}, reasoning_brief}, ...]}
{"tick": N, "event_type": "movement", "actors": [...], "from": "zone", "to": "zone", ...}
{"tick": N, "event_type": "zone_entered", "zone": "...", "actors": [...], ...}
{"tick": N, "event_type": "speech", "actor": "pX", "text": "...", "visibility": [...], ...}
{"tick": N, "event_type": "tick_completed", ...}
```

dm_query → player_response пары обязательны для значимых тиков.

# Part 9: Protocol — No Silent Omission of Model Layers

## Главный принцип

Это симуляционный стенд, не генератор историй. Silent omission любого объявленного слоя модели — запрещён.

Если слой, механизм или причинный шаг не был реально вычислен в этом прогоне — DM должен заявить об этом явно. DM никогда не представляет результаты как произведённые слоем который не был использован.

## Обязательная прозрачность

Для каждого прогона и желательно для каждого тика — явно отслеживать статус всех важных слоёв:

- **used** — использовался
- **updated** — обновлялся
- **skipped** — пропущен
- **reason** — причина

Минимально применяется к:
- world_cues / sensing
- attention selection
- appraisal
- state_update
- memory / imprint activation
- **relationship / affinity / social dynamics**
- action_pulls

## Жёсткий запрет

DM не может:
- молча пропустить слой и раскрыть это только если спросят
- выводить причинные объяснения через слой который не был вычислен
- описывать социальные или реляционные последствия как произведённые моделью если affinity/relationship логика была пропущена
- имитировать стиль предыдущих симулированных результатов без сохранения реального базового процесса

## Если слой пропущен

Если слой пропущен, DM должен:
1. явно пометить его как skipped
2. объяснить почему пропущен
3. ограничить выводы чтобы они не зависели от этого слоя
4. пометить прогон как partial для оценки полной модели если пропущенный слой был критичен

## Валидность прогона

Каждый прогон должен включать метку валидности:

- `valid_for_model_testing: true` — полностью валиден
- `valid_for_model_testing: partial` — частично валиден
- `valid_for_model_testing: false` — не валиден

И краткую причину.

## Формат model_layers в summary

```json
{
 "model_layers": {
   "world_cues": { "used": true, "updated": true },
   "attention": { "used": true, "updated": true },
   "appraisal": { "used": true, "updated": true },
   "state_update": { "used": true, "updated": true },
   "memory_activation": { "used": true, "updated": true },
   "affinity": {
     "used": false,
     "updated": false,
     "reason": "not computed in this run"
   },
   "relationship_dynamics": {
     "used": false,
     "updated": false,

   },
   "action_pulls": { "used": true, "updated": true }
 },
 "valid_for_model_testing": "partial",
 "validity_reason": "social-dynamics-related conclusions are limited because affinity was skipped"
}
```
